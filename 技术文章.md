# LightPick 技术解析：以 CRDT 画布为运行时的多智能体视频创作系统

## 摘要

LightPick 不是一个单纯的 AIGC 前端，也不是把若干模型 API 包成按钮的工具站。当前代码呈现的是一套面向视频创作的协同运行时：人、浏览器、CLI、云端 Agent、本地 Agent 都围绕同一份 Loro CRDT 画布工作；生成任务由 Cloudflare Durable Objects 和 Workflows 接管；媒体资产进入 R2，元数据进入 D1；模型选择、引用校验、节点创建、自动布局和任务触发被收敛到共享类型包中。

这套架构的核心价值在于：画布既是用户界面，也是系统状态、任务编排和 Agent 可操作的事实源。与传统工作流画布或通用 Agent 框架相比，LightPick 更强调多人实时编辑、可恢复生成流程、可视化资产谱系，以及跨 Web、CLI、SDK 的一致接入模型。

## 这个系统做了什么

LightPick 面向“多人 + 多智能体”的视频创作场景，提供从创意节点、文生图、图生视频、文本生成、音频生成，到 Remotion 渲染输出的完整链路。用户在画布上创建文本、图片、视频、音频和 ActionBadge 节点；点击执行后，系统会生成 pending 资产节点，由后端 Durable Object 扫描并提交到 Cloudflare Workflow，最终把生成结果写入 R2/D1，并通过 Loro 广播回所有在线客户端。

从代码看，系统可以拆成五层：

1. **Web 前端层**：`apps/web` 提供 React/Vite 前端和 Cloudflare Worker 入口，负责静态资源、Better Auth、网关代理和浏览器会话。
2. **API 与实时运行时层**：`apps/api-cf` 基于 Hono，承载 `/api/v1/*`、`/sync/*`、`/agents/*`、资产、缩略图、上传和插件扩展路由。
3. **协同画布层**：`ProjectRoom` Durable Object 维护 Loro 文档、WebSocket 同步、presence/activity sideband、pending task 扫描和任务轮询。
4. **生成工作流层**：`GenerationWorkflow` 根据任务类型和模型 ID 选择 provider，在 Cloudflare Workflows 的 durable step 中调用 Google、fal.ai、Kling、OpenAI、Remotion 等能力。
5. **共享领域模型层**：`packages/shared-types` 定义 Canvas API、节点 schema、模型卡、引用能力、任务类型和 Loro 客户端，供 Web、CLI、Agent 共用。

整体链路可以概括为：

```text
浏览器/CLI/Agent 写入 Loro 画布
  -> ProjectRoom 接收二进制 CRDT update
  -> 扫描 status=pending 的资产节点
  -> NodeProcessor 创建 GenerationWorkflow
  -> Workflow 选择 provider 并执行 durable steps
  -> 结果上传 R2，资产元数据写入 D1
  -> 通知 ProjectRoom 更新节点状态和 assetId
  -> Loro 广播回浏览器、CLI、Agent
```

## 核心实现拆解

### 1. 画布是唯一运行时状态

LightPick 的一个关键设计是“Loro 是画布唯一真相”。节点、边、任务状态、pending 资产节点都在 CRDT 文档中传播；D1 不重复维护画布字段，而是保存项目、用户、API token、资产元数据、运行时信息等关系型数据。

`ProjectRoom` 是这层的核心。它在 WebSocket 升级时加载快照和增量日志，给新客户端发送 Loro snapshot；后续浏览器或 CLI 发来的 ArrayBuffer 被串行导入 `doc.import()`，再广播给其他客户端。为了避免 CRDT 导入并发导致状态异常，代码通过 message queue 串行处理 update；同时将远端导入和本地提交都写入 append-only update log，并在达到阈值后 compact 成新快照。

协同体验不是额外通道，而是同一条 WebSocket 的 sideband JSON 消息。二进制消息用于 Loro CRDT，同一连接上的字符串消息用于 presence、activity、custom action 注册、任务完成通知等。这让协作可见性和画布状态保持在同一个房间模型里。

### 2. Canvas API 收敛业务语义

`packages/shared-types/src/canvas-ops.ts` 中的 `Canvas` 类是 Web、CLI、Agent 共同使用的业务入口。它不是简单的 CRUD wrapper，而是把节点创建、自动布局、边插入、状态查询和生成触发收敛成稳定方法：

```ts
canvas.listNodes(type, parentId)
canvas.readNode(nodeId)
canvas.createNode(nodeId, type, data, position, parentId)
canvas.createLinkedNode({ sourceNodeId, nodeType, data })
canvas.updateNode(nodeId, updates)
canvas.deleteNode(nodeId)
canvas.executeGeneration(nodeId, generateId)
```

其中 `executeGeneration()` 尤其重要。它会检查目标节点是否为生成类 ActionBadge，抽取 prompt，解析 `modelId/modelParams`，基于模型卡校验引用输入，然后调用 `buildPendingAssetNode()` 创建 pending 资产节点，并自动插入边和布局位置。也就是说，“执行生成”不是散落在前端、CLI、Agent 中的重复流程，而是一段共享领域逻辑。

这个设计直接降低了多入口系统的漂移风险。浏览器点击、CLI 命令、SupervisorAgent 工具调用，都可以走同一个 Canvas 语义，而不是各自拼 node data。

### 3. 模型卡不是配置表，而是能力契约

`packages/shared-types/src/models.ts` 中的 `MODEL_CARDS` 统一描述模型 ID、provider、输出类型、参数 schema、默认参数和输入约束。`model-capabilities.ts` 再把模型卡归一化为统一的 `Capability`：

```ts
capability(card).ref.image.accepts
capability(card).ref.video.max
capability(card).requiresPrompt
capability(card).promptModalities
```

这解决了多模态生成系统里很常见的问题：不同模型是否接受图片、视频、音频、文本引用？是 start/end frame 还是多图参考？引用数量上下限如何校验？默认模型如何根据上游节点类型选择？

LightPick 没有把这些逻辑写死在 UI 组件里，而是通过 `validateRefs()`、`partitionRefs()`、`pickDefaultModel()` 统一派生。前端可以用它决定按钮是否展示、参数面板如何渲染；后端可以用它验证任务输入；Agent 可以用它选择可执行模型。这比“provider if/else + UI 特判”的模型接入方式更可维护。

### 4. 生成任务由 Workflow 承担可恢复执行

生成不是请求线程里同步完成的。`NodeProcessor` 扫描 Loro 中的 pending 节点，设置 `pendingTask` 作为乐观锁，然后通过 `startGeneration()` 创建 Cloudflare Workflow。`startGeneration()` 是唯一调度入口，它会先执行插件的 `beforeGenerationStart` hook，再调用 `GENERATION_WORKFLOW.create()`，因此可以把额度、风控、限流等失败前置到毫秒级反馈。

`GenerationWorkflow` 自身只承担平台调度职责：

- 创建 `GenerationContext`
- 通过 `resolveProvider()` 选择 provider
- 执行插件 before/after/failure hook
- 调用 provider 的 `execute(ctx)`
- 成功或失败都记录事件，并将结果通知回 ProjectRoom

具体模型调用由 provider 负责。`resolveProvider()` 根据 `params.type` 和模型 ID 路由到 Google image/video/text、fal image/video、Kling image/video、Gemini TTS、Remotion render、custom action 等实现。`GenerationContext` 则封装了 durable step、R2 读写、asset probe、D1 asset 创建和 Loro 节点通知。

这体现了一个清晰的 DIP 设计：平台不规定每个模型的 step graph，只提供稳定的上下文能力；provider 可以按自己的 API 形态组织步骤，但输出都回到统一的资产和节点协议。

### 5. SupervisorAgent 是画布协作者，不是外部聊天机器人

`SupervisorAgent` Durable Object 每个实例维护一份本地 Loro replica，并通过内部 WebSocket 接入 `ProjectRoom`。浏览器连接它进行 AI chat，它再通过 canvas tools、workflow tools、timeline tools 操作同一份画布。

这和常见“聊天窗口调用后端 API”不同。Agent 的上下文不是隐藏在 prompt 里的摘要，而是实时同步的 CRDT 画布。它可以读取节点、创建生成节点、触发工作流、等待生成结果、把工具返回的图片重新注入模型消息，并且在 ProjectRoom 连接中断时缓存 pending broadcasts，重连后用 CRDT merge 修复潜在丢写。

代码里还处理了 Durable Object hibernation 后的身份恢复、dangling tool call 修复、首包超时、总轮次超时、房间 WebSocket 自愈等问题。这说明它不是 demo 级 agent，而是在 Cloudflare 运行模型下为长连接、长任务和可恢复协作做了工程兜底。

## 与同类框架有什么不同

### 相比通用工作流画布

很多工作流画布强调“节点图 = 任务 DAG”，执行时把图编译成一次性 pipeline。LightPick 的画布更接近“多人实时工作台”：节点既可以是 prompt、资产、生成动作、分组，也可以承载任务状态、引用关系、理解结果和编辑结果。它不是只为一次运行服务，而是长期保存创作过程。

关键差异是：LightPick 的图是 CRDT 文档，天然支持多人、CLI、Agent 同时编辑；任务执行是从画布状态中被动发现 pending 节点，而不是由单个前端提交一份静态 DAG。

### 相比通用 Agent 框架

通用 Agent 框架通常把工具调用和状态保存在会话内部，用户能看到的是聊天 transcript。LightPick 把 Agent 的行为落到可视化画布中：Agent 创建的节点、引用的资产、执行中的任务、失败原因和最终结果都对人类协作者可见。

这会带来更强的可审计性。人可以接手 Agent 的中间产物，CLI 可以继续操作同一节点，另一个 Agent 也能基于当前画布状态协作，而不是依赖某个 agent memory 的隐式上下文。

### 相比模型 API 聚合站

普通模型聚合站往往把差异封装成一个统一表单，牺牲了多模态引用和模型能力的精确表达。LightPick 通过模型卡描述不同 provider 的参数、默认值、输入模态、引用数量、最大运行时，再通过 capability 层归一化给 UI、后端和 Agent 使用。

因此它可以同时支持文本、图像、视频、音频模型，也能表达 start/end frame、多参考图、视频引用、音频引用等差异，而不是把所有模型降维成一个 prompt textarea。

### 相比传统 Web 应用后端

传统 Next.js API + 数据库 CRUD 模式不适合长任务、多客户端实时编辑和边缘可恢复执行。LightPick 把 `/api/v1/*` 放在 `api-cf` 的 Hono Worker 中，把 WebSocket 房间放入 Durable Objects，把长任务放入 Workflows，把媒体放入 R2。这种 Cloudflare-first 拆分让每个运行时承担自己的职责：

- Web Worker 负责 SPA、Better Auth 和网关代理。
- Hono Worker 负责 REST、资产、插件和路由。
- Durable Object 负责单项目串行协同状态。
- Workflow 负责可重试、可恢复的长任务。
- Container 负责 Remotion 渲染这种 Node/Chromium 重任务。

## 如何接入和使用

### 1. 本地启动

仓库提供了 Makefile 封装常用命令：

```bash
make install
make db-local
make dev
```

默认开发模式会启动 Web 前端和 `api-cf`。如果需要完整网关链路，可以使用：

```bash
make dev-gateway-full
```

需要注意的是，所有新的 `/api/v1/*` 接口都应添加到 `apps/api-cf/src/routes/v1/`，并在 `apps/api-cf/src/routes/v1/index.ts` 注册。网关会拦截 `/api/v1/*` 到 `api-cf`，不要在 Next.js 或 Web app 下新增同路径 API。

### 2. 浏览器接入

浏览器侧主要通过网关访问：

- `/`：Web SPA
- `/api/better-auth/*`：Better Auth session
- `/api/v1/*`：鉴权后的 REST API
- `/sync/:projectId`：连接 ProjectRoom 的 Loro CRDT WebSocket
- `/agents/supervisor/:projectId:threadId`：连接 SupervisorAgent 的聊天 WebSocket
- `/assets/*`、`/upload/*`、`/thumbnails/*`：资产读写和缩略图

网关会对 `/api/v1/*` 解析 Better Auth session 或 `clsh_*` API token，并注入 `x-user-id` 给下游服务。浏览器只需要保持登录态，CLI/Agent 则使用 API token。

### 3. CLI 接入

CLI 的入口是 `packages/cli`，命令名为 `lightpick`。典型流程如下：

```bash
lightpick auth login
lightpick auth status
lightpick projects list
lightpick canvas list --project <project-id>
lightpick canvas add --project <project-id> --type image_gen --label "海报" --content "cinematic poster"
lightpick canvas execute --project <project-id> --node <node-id>
```

CLI 不是普通 REST wrapper。它会通过 `LoroSyncClient` 连接 `/sync/:projectId`，拿到初始 CRDT snapshot 后，在本地 `Canvas` 上执行同一套节点操作，再把本地 update 通过 WebSocket 发回 ProjectRoom。对于连续操作，还支持 daemon 模式，避免每条命令都重新建立同步连接：

```bash
lightpick canvas connect --project <project-id>
lightpick canvas list --project <project-id> --json
lightpick canvas disconnect --project <project-id>
```

### 4. TypeScript 集成

外部 TypeScript 工具可以直接复用 `@lightpick/shared-types` 的 Loro 客户端和 Canvas API：

```ts
import WebSocket from "ws";
import { LoroSyncClient } from "@lightpick/shared-types";

const client = new LoroSyncClient({
  serverUrl: "ws://localhost:8788",
  projectId: "<project-id>",
  token: "clsh_xxx",
  clientType: "cli",
  WebSocket,
});

await client.connect();

const result = client.canvas.createNode(
  crypto.randomUUID().slice(0, 8),
  "image_gen",
  {
    label: "产品主视觉",
    content: "A clean studio shot, red accent, cinematic lighting",
  },
);

await client.disconnect();
```

如果要触发生成，先创建生成类 ActionBadge，再调用：

```ts
const run = client.canvas.executeGeneration(nodeId, () =>
  crypto.randomUUID().slice(0, 8),
);
```

`executeGeneration()` 会自动创建 pending 资产节点。ProjectRoom 收到 update 后，后端会继续接管工作流调度。

### 5. 新增模型或 provider

接入新模型建议按三步做：

1. 在 `packages/shared-types/src/models.ts` 增加 `MODEL_CARDS` 项，描述 `id`、`provider`、`kind`、参数、默认值、输入模态和运行时上限。
2. 如果是新 provider，在 `apps/api-cf/src/generation/providers/` 实现 `GenerationProvider`，通过 `GenerationContext` 完成 step、R2、D1、ProjectRoom notify。
3. 在 `apps/api-cf/src/generation/registry.ts` 的 `resolveProvider()` 中把模型 ID 或任务类型路由到新 provider。

这样前端、CLI、Agent 和后端校验都能从模型卡自动获得能力信息，避免重复写输入约束。

### 6. 新增 API

新增公开 REST API 时，应放在 `apps/api-cf/src/routes/v1/` 下，并注册到 `v1Routes`。请求鉴权由网关或 `api-cf` middleware 解析成 `x-user-id`，业务 route 可以只关心用户身份和 Zod 校验。

示例职责边界：

- `/api/v1/projects`：项目 CRUD 和房间消息。
- `/api/v1/assets`：资产元数据和签名 URL。
- `/api/v1/runtimes`：本地 runtime/daemon 连接。
- `/api/v1/crew`：多 Agent/crew 相关能力。

### 7. 自定义 action 和插件扩展

LightPick 留了两类扩展口：

第一类是 canvas action。`@lightpick/action-sdk` 定义了 `ActionManifest`、`ActionRequest`、`ActionResponse`，用于把外部 Worker 或本地能力接成画布动作。Python SDK 也提供了 `@action(...)` 装饰器和 `LightPickAgent` 形态，用来注册本地 action manifest，并通过 WebSocket sideband 与 ProjectRoom 交互。当前 Python 侧对 Loro task 解析仍保留 TODO，因此更适合作为本地 action 接入骨架或后续扩展基础。

第二类是 `api-cf` plugin。`createApp({ plugins })` 可以注册 auth、generation、assets、routes hook。生成前置 hook 可以在 Workflow 创建前做额度/限流/密钥选择，失败会直接反馈到调用方；Workflow 内 hook 则适合做真实调用前后的审计、计费或观测。

## 工程价值总结

LightPick 的专业性不在于把很多模型 API 串起来，而在于它把复杂创作系统中的几个高风险边界做成了稳定抽象：

- **状态边界清晰**：Loro 管画布，D1 管资产和业务元数据，R2 管媒体 blob，不重复维护 `src` 等易漂移字段。
- **执行边界清晰**：ProjectRoom 负责发现 pending 节点和协同广播，Workflow 负责长任务恢复，provider 负责模型调用细节。
- **接入边界清晰**：Web、CLI、Agent 共享 `Canvas` 和模型能力，不各自实现一套生成流程。
- **扩展边界清晰**：模型通过 `MODEL_CARDS + provider` 接入，业务插件通过 hook 接入，自定义 action 通过 manifest/request/response 接入。
- **协作边界清晰**：人和 Agent 操作同一份 CRDT 画布，过程可见、状态可接管、结果可追踪。

如果用一句话概括：LightPick 把“视频创作画布”从一个前端交互组件提升成了一个可协同、可恢复、可扩展的边缘运行时。它的核心竞争点不是单个模型效果，而是让创作过程本身成为多人、多 Agent、多工具都能安全操作的共享系统。
